# 프로젝트 작업 지침

이 프로젝트는 **전력산업 교육용 게임**이다.
그래서 우리의 개발 작업도 **실제 한전·전력거래소(KPX) 현장의 안전작업 절차를 메타포로** 사용한다.
규칙을 외우는 게 아니라, 코드 작업을 하면서 자연스럽게 전력 현장 용어를 익히는 게 목적이다.

---

## 🔌 메타포: 코드 작업 = 선로 작업

| 코드 작업 | 전력 현장 |
|---|---|
| 메인이 직접 파일 수정 | **활선작업** (Live-line work) — 전기 흐르는 상태에서 작업 |
| `worktree` 격리 후 수정 | **휴전작업** (De-energized work) — 전기 끊고 안전하게 작업 |
| 같은 파일을 두 에이전트가 동시 수정 | **2팀이 같은 선로에서 활선작업** → 사고 |
| `git stash` / commit으로 상태 정리 | **접지(Grounding)** — 잔류전압 방전 |
| `git status` 확인 | **검전(Voltage Check)** — 무전압 상태 확인 |
| 에이전트에게 수정 범위 명시 | **작업표찰(Tag-out)** — "이 구간만 작업" 표시 |
| 사용자 동의 받기 | **작업허가(Work Permit)** — 급전소 승인 |
| diff 검토 후 머지 | **송전 재개(Re-energize)** — 작업 완료 후 통전 |
| 롤백 | **재폐로(Reclosing)** — 차단 후 재투입 |

## ⚡ 서브에이전트 병렬 호출 = 다중작업 협의

### 작업 전 9단계 절차 (한전 실무 절차 그대로)

병렬 에이전트를 호출하기 전에 아래 흐름을 따른다:

1. **작업계획서 작성** → 사용자에게 무엇을·어디서·왜 수정할지 한 줄 보고
2. **휴전 요청 / 활선 판단** → worktree 격리 필요한지 결정
3. **검전 (Voltage Check)** → `git status`로 unstaged 변경 확인
4. **접지 (Grounding)** → 변경사항이 있으면 commit 또는 stash로 안전화
5. **작업표찰 (Tag-out)** → 에이전트 프롬프트에 **수정 가능한 파일 범위 명시**
6. **작업 수행 (Execution)** → `Agent` 호출
7. **작업종료 확인** → 결과 diff 검토
8. **접지 제거** → 머지 / 통합
9. **송전 재개** → main 브랜치 반영

### 활선 vs 휴전 판단표

| 상황 | 작업 방식 | 이유 |
|---|---|---|
| 읽기·조사만 | **활선 OK** (병렬) | 전기 안 건드림 |
| 서로 다른 파일 쓰기 | **활선 OK** (병렬, 표찰 명시) | 다른 선로 작업 |
| 같은 파일 동시 쓰기 | **휴전 필수** (worktree) | 동일 선로 = 사고 위험 |
| `package.json`·`vite.config.js` 등 모선 설비 | **휴전 + 단독 작업** | 모선 정전은 전 구역 영향 |
| `git commit`·`push` | **직렬, 급전소 단일 지령** | 두 명이 동시에 차단기 못 조작 |

### N-1 신뢰도 기준 (Reliability Criterion)

실제 계통 운영의 N-1 원칙: **설비 하나가 고장 나도 정전이 없어야 한다.**
우리 코드: **에이전트 하나가 실패해도 다른 작업이 망가지지 않게** → 격리·독립성 확보 (worktree가 이를 보장).

## 🚨 금지 패턴 (= 사고 사례)

- ❌ **무허가 활선작업**: 같은 파일을 수정하는 2개 이상 에이전트를 worktree 없이 병렬 호출
- ❌ **모선 단독작업 위반**: 한 에이전트가 `package.json`·`vite.config.js`·공유 설정 수정 중 다른 에이전트 병렬 실행
- ❌ **이중 급전지령**: 병렬 에이전트들이 모두 `git add`/`git commit` 수행
- ❌ **무표찰 작업**: "알아서 잘 해줘" 식 광범위 프롬프트로 병렬 호출 — 수정 범위가 불명확하면 표찰 없는 작업과 같음

## 🎓 게임 콘텐츠에도 같은 절차 반영하기

코드 작업 절차를 게임 미션으로도 녹일 것:
- **튜토리얼 미션**: "변압기 점검 — 휴전 요청 → 검전 → 접지 → 작업 → 송전" 9단계 체험
- **이벤트**: "활선상태에서 작업 시도 → 감전 사고 → 미션 실패" 페널티
- **N-1 챌린지**: "송전선 1회선 고장 — 다른 선로로 우회 급전 성공시키기"
- **급전소 역할극**: 플레이어가 작업허가를 발급하는 디스패처 시점

작업 절차를 비유로 쓰는 이유: **사용자(개발자)가 직접 절차를 체험해야, 게임에 그 절차를 자연스럽게 녹일 수 있다.**

## 📋 작업 시작 전 사용자에게 확인할 상황

다음에 해당하면 에이전트 호출 전 한 줄로 보고 후 동의 받기 (= 작업허가 발급):

- 동일 파일을 수정할 가능성이 있는 에이전트를 2개 이상 띄울 때
- worktree 격리를 사용할 때 (특히 unstaged 변경이 있는 경우)
- 에이전트가 git commit/push를 수행할 때
- 모선급 공유 파일(`package.json`, `vite.config.js`, `src/App.jsx`)을 수정할 때
